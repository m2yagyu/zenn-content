# Zenn記事執筆ワークフロー

このリポジトリ(zenn-content)で新しい記事を書くときの標準手順。
執筆スタイル・テーマの詳細な方針は必ずリポジトリ直下の `CLAUDE.md` を読んで最優先する(テーマは過去に複数回改定されているため、このSKILL.mdの記述より `CLAUDE.md` の内容を信頼すること)。

## 1. テーマとヒアリング
- テーマ・対象読者・分量感をユーザーに確認する
- 記事の内容や難易度に大きく影響する技術的な分岐(使うAPI/モデル、有料か無料か、依存ライブラリなど)があれば、本文を書き始める前に必ずユーザーに確認する。書いてから手戻りさせない

## 2. アウトラインの提示
- 見出し構成だけを先に提案し、ユーザーの承認を得てから本文を書き始める
- `CLAUDE.md` のスタイル方針(冒頭2行で得られる結果を明示、動くコードを先に見せて理論・数式は後半、数式の前に日本語での直感的な説明を挟む)を反映した構成にする

## 3. コードの動作検証(必須・省略しない)
- 記事に載せるコードは、公開前に必ずこのセッション内で実際に実行して動作確認する。検証できていないコードは記事に含めない
- 検証のために依存パッケージのインストールが必要な場合は、スクラッチパッド配下に使い捨ての仮想環境を作って検証し、終わったら必ず `rm -rf` で削除する(通常のPython環境やこのリポジトリを汚さない)
  ```bash
  python3 -m venv <scratchpad>/verify_venv
  source <scratchpad>/verify_venv/bin/activate
  pip install -q <必要なパッケージ>
  # 検証後
  rm -rf <scratchpad>/verify_venv
  ```
- Google Colab向けのAIデモを書く場合は、ローカルへの大容量モデルダウンロードより、Hugging Face推論APIのような「ダウンロード不要」な方法を優先する。過去にHugging Face Hub上のモデルダウンロードがColab環境で停止し(xet/CDN起因)、初学者には対処不能な複雑さになった教訓があるため
- 記事中の会話ログ・実行結果は、検証で実際に得られた本物の出力をそのまま使う。都合よく作文しない

## 4. 図解
- 図が必要な場合はmatplotlib(データの可視化)またはmermaid(処理の流れ図)で生成する
- mermaidのフローチャートは `graph TD` の一直線構成にし、双方向の矢印やループを作らない(往復矢印があるとレイアウトが詰まって文字が小さくなり読みにくくなることが実際にあった)
- matplotlibで日本語ラベルを使う場合はフォント(例: Hiragino Sans)を明示指定し、文字化けしないことを確認してから保存する
- 画像はリポジトリ直下の `/images/` に保存し、記事内では絶対パス `/images/xxx.png` で参照する(相対パスは不可。GitHub連携で自動的にZennへアップロードされる)

## 5. 記事ファイルの作成
```bash
npx zenn new:article --slug <slug> --title "<タイトル>" --type tech --emoji "<絵文字>"
```
- frontmatterの `topics` を内容に合わせて整える
- 本文を書き終えるまで `published: false` のままにしておく

## 6. ローカルプレビューと公開判断
- 本文完成後、`npx zenn preview` をバックグラウンドで起動し、`curl` でページが200を返すか(構文エラーがないか)だけ機械的に確認する
- 図やmermaidの実際の見た目・レイアウトは機械的に確認できないため、ユーザー自身に `npx zenn preview` をブラウザで開いて確認してもらい、OKをもらってから次に進む
- 問題なければコミット・プッシュする(この時点では `published: false` のまま)
- ユーザーが明示的に「公開する」と言ってから、frontmatterの `published` を `true` に変更し、再度コミット・プッシュする。ユーザーの明示的な合意なしに公開状態を変えない

## 7. コミット時の注意
- このリポジトリのローカルgit identity(`m2yagyu <mitsu.yagyu@gmail.com>`)を使う。グローバル設定は変更しない
- APIキーやトークンなど秘密情報は `.env` に書かせて読み込む(チャットに直接貼らせない)。もし誤ってチャットに貼られた場合は、検証後に失効(revoke)して新しいものを発行するようユーザーに伝える
